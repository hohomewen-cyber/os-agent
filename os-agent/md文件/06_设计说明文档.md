# 设计说明文档

## 1. 整体架构

系统由六层组成：

1. 交互层：Streamlit 页面负责文本/语音输入、状态展示和执行反馈。
2. Agent 决策层：千问模型负责意图理解、命令生成、复杂任务拆解和反馈总结。
3. 执行控制层：安全规则对命令进行风险评估、前置检查、确认与拦截。
4. 环境感知层：自动识别远程系统、包管理器、当前用户与权限状态。
5. 环境执行层：通过 SSH 在真实 Linux 服务器执行命令并返回结构化结果。
6. 审计持久层：本地 JSON 文件保存审计日志，支持刷新恢复、导出与清空。

## 2. 关键设计取舍

### 2.1 采用 Web 交互而非纯 CLI

原因：

- 更适合展示风险提示、执行过程和多轮对话历史
- 便于引入语音录音能力
- 更符合“AI 作为主要交互入口”的探索方向

### 2.2 采用规则风控 + 大模型决策的双层机制

原因：

- 模型负责灵活理解自然语言
- 本地规则负责兜底高风险识别
- 避免完全依赖模型造成风险边界不稳定

### 2.3 语音能力采用独立辅助模块

原因：

- 避免侵入现有执行主链
- 只在输入侧新增能力，不改变既有任务规划与执行逻辑
- 便于单独测试语音识别链路

### 2.4 结构化执行结果作为唯一真值

原因：

- 避免依赖字符串是否包含 `error` 来猜测命令成功与否
- 让 UI 展示、AI 反馈、审计日志和终端镜像基于同一执行结果
- 为执行前状态检查和执行后状态验证提供统一承载结构

## 3. 题目能力映射

### 3.1 基础能力

- 自然语言查询磁盘、内存、端口
- 文件与目录检索
- 普通用户创建与删除
- 过程与结果自然语言反馈

### 3.2 进阶能力

- 高风险命令识别
- 中高风险预警与二次确认
- 安全判定逻辑说明

### 3.3 探索能力

- 多轮上下文对话
- 多步连续任务处理
- 语音输入
- 去命令行化交互体验

### 3.4 操作环境感知与决策

- 自动探测远程主机操作系统类型、发行版和包管理器
- 自动探测当前用户、是否 root、是否可用 sudo
- 环境信息写入 `executor.py` 的 `self.env_info`
- Agent system prompt 动态注入环境信息，减少跨环境误判

## 4. 语音输入设计

### 4.1 链路

1. 浏览器录音
2. 音频上传到 Streamlit
3. 后端调用千问 ASR 模型
4. 提取转写文本
5. 前端脚本自动写入聊天输入框

### 4.2 设计目标

- 不替换原有聊天输入能力
- 不改动现有任务执行逻辑
- 录音结束后立即转写
- 转写完成后自动回填，保证用户可见、可继续编辑

## 5. 行为设计逻辑

- 先由模型生成执行计划
- 再由执行器二次检查风险，并执行必要的前置状态检查
- 中高风险触发确认或拒绝
- 执行后进行状态验证，再根据统一结果生成解释性反馈
- 最近历史写入上下文，以支持连续任务

## 6. 闭环增强点

### 6.1 执行前状态检查

- 删除文件前检查目标是否存在
- 修改权限/属主前检查目标是否存在
- 结束进程前检查 PID 或匹配进程是否存在
- 删除用户前检查用户是否存在

### 6.2 执行后状态验证

- 安装软件后验证包是否已安装
- 删除文件后验证目标是否已删除
- 服务命令后验证服务状态是否符合预期
- 创建/删除用户后验证用户状态是否符合预期

### 6.3 文档解析策略

- AI 抽取作为主链，负责理解多格式文档中的 Linux 命令和 C 代码
- 规则识别作为兜底，保证在无 API Key 或模型失败时仍可解析基础内容
- 解析诊断信息回显到页面，便于答辩和排障

## 7. 限制条件

- 依赖千问 API 与网络连通性
- 依赖可访问的 Linux 服务器
- 语音输入当前实现为“录音 -> 转写 -> 文本回填”，不包含语音播报输出

## 8. 后续可扩展方向

- 语音输入后自动提交
- 更多 Linux 发行版和包管理器适配
- 更细粒度的命令审计与回滚策略
